Executive Summary

VITARA's design system must work at two scales: a marketing website that positions the product for users who have never heard of it, and a mobile health app that users interact with every single day. The marketing site uses Cormorant Garamond for editorial authority and Plus Jakarta Sans for clarity — a combination that reads as premium and considered, not clinical. The mobile app uses the native type system (SF Pro on iOS, Google Sans on Android) because health data is a daily-use interface: native typography scales, renders accessibly, and respects user system settings including Dynamic Type. The binding thread between both surfaces is a restrained palette anchored in deep forest green (#1A5C3F), warm cream (#F8F7F3), and clean white — colors that feel warm and trustworthy without triggering the "medical app blue" associations of every other health product on the market.

Design Philosophy

Principle 1 Culturally Neutral UI

The interface must feel appropriate for every background represented in our user base. This means avoiding icons, illustrations, or color meanings that carry strong cultural connotations in specific contexts. No food photography of any single cuisine in default states. No skin tone or body type assumptions in avatar/illustration systems. The UI's job is to surface the user's own cultural context — not impose one.

Principle 2 Clarity Over Cleverness

Health data must be immediately readable. No ambiguous chart axes. No unlabeled icons for core navigation. When a user sees their HRV trend, they should understand what it means without tapping for a tooltip. Labels, units, and context live next to data by default — not hidden behind a "learn more" tap. Clever animations and interactions are earned, not applied by default.

Principle 3 Data-Forward Display

Health data should be displayed as information, not decoration. Numbers need context: show the trend arrow alongside the value, show the 7-day range alongside today's reading, show what "good" looks like relative to the user's own history. A sleep score of 72 means nothing without knowing it was 68 yesterday. The design system encodes this: health metric components always include trend and reference, never just the current value.

Principle 4 Accessibility First

WCAG AA is the minimum bar, not the aspiration. Touch targets at 44px minimum, contrast at 4.5:1 for all body text, 3:1 for large text and UI components. Screen reader labels on all health data readouts: "Heart rate: 68 beats per minute, down 3 from yesterday." Reduce-motion preferences are respected at the system level — no CSS or Flutter animation overrides the user's OS setting.

Principle 5 Premium but Warm

Clinical apps feel cold. Wellness apps feel saccharine. VITARA should feel like a thoughtful, trusted advisor — the visual equivalent of a well-designed notebook, not a hospital form or a motivational poster. The design restraint (limited color, generous whitespace, serif headlines) creates the premium register; the warm cream background and the specificity of culturally accurate content create the warmth.

Cultural Neutrality in Practice
Specific decisions required by this principle: any default illustrations showing food must be abstract or minimal, not depicting a specific cuisine; person illustrations (onboarding, empty states) must include diverse skin tone representation; icon choices should avoid double meanings across cultures (the thumbs-up gesture is offensive in some South Asian contexts; certain color meanings differ by culture — red means danger in Western UI but celebration in South Asian contexts). When in doubt, use text labels rather than icons for anything culturally interpretable.

Health Data Display
The principle that numbers need context applies directly to typography: never display a health metric at xl/2xl/3xl scale without also displaying a trend indicator at sm/base scale immediately adjacent. A large "72" sleep score with a small "+4 from yesterday" and a small "/ 100" reference is far more meaningful than the number alone. This pairing is mandatory in the HealthMetricCard and ProgressRing components — it is baked into the component API, not optional.

Motion Principles

Duration TokenValueUse Case
duration.micro150msButton press feedback, checkbox toggle, switch toggle, icon swap
duration.standard250msCard appear/disappear, input focus border transition, chip selection, tab switch
duration.complex400msScreen transitions (page route), progress ring fill, onboarding step advance, modal sheet presentation
duration.slow600msPlan generation loading screen states, daily brief reveal

Easing: Use Curves.easeInOut as the default for all transitions. Use Curves.easeOut for elements entering the screen (feels like arrival). Use Curves.easeIn for elements leaving the screen. Never use linear easing for UI transitions.

Reduce motion: Check MediaQuery.of(context).disableAnimations at the app root. When true, replace all AnimatedWidget and transition animations with instant state changes. Health data progress rings still animate their value change (functional feedback) but at 0ms duration — the ring jumps to the correct value rather than sweeping.

No decorative animation: VITARA does not use looping animations, particle effects, or ambient motion. Animations are always in response to a user action or a data state change — never running unsolicited in the background.

Accessibility Requirements

  • Contrast: WCAG AA minimum — 4.5:1 for all body text (16px and below), 3:1 for large text (18px+ or 14px+ bold) and UI components (borders, icons). Test in both light and dark mode.
  • Touch targets: Minimum 44×44 logical pixels for all interactive elements. Use GestureDetector or InkWell with a padding wrapper if the visual element is smaller.
  • Dynamic Type (iOS): All text must respect the user's system font size setting. Use Flutter's TextScaleFactor or rely on Material Typography which respects it automatically. Test at Accessibility Extra Large font size — layouts must reflow gracefully, not overflow.
  • Screen reader support: Every health metric readout requires a Semantics widget with a complete spoken representation: value, unit, trend, and context. Example: Semantics(label: 'Sleep score: 74 out of 100, up 3 from last night', child: SleepScoreWidget(...))
  • Focus management: After a modal sheet opens, focus must move to the first interactive element in the sheet. When it closes, focus returns to the element that triggered it.
  • Error announcements: Form validation errors are announced to the screen reader when they appear, not only displayed visually. Use SemanticsService.announce() or a live region equivalent.
  • Color-independent information: Trend arrows never communicate direction through color alone — the arrow direction (up/down/flat) and a text label ("+3", "-2", "stable") always accompany any color-coded trend indicator.
