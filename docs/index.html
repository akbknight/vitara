v0.1 · Customer Discovery

Every document in this collection reflects the current best understanding of what VITARA needs to be — shaped by the startup strategy process, feedback from Professor Ulstrup, advice from Nilu, and the ITEC-617 pitch at American University.

We are in customer discovery. Every feature listed in the PRD is a hypothesis to test, not a requirement already committed. Every technical decision is a proposal to validate, not a contract already signed. The purpose of writing this down is to have a clear, contestable baseline — something concrete enough to be wrong about.

Documents are updated as discovery interviews reshape our understanding. The version number on each document reflects its stage in that process.

The stack at a glance

What we're
planning to build.

Full details in the Technical Architecture document. These are the key decisions and their rationale.

Layer Technology Rationale
Mobile Flutter 3.x Single codebase for iOS + Android. Dart performance for health data rendering. HealthKit + Health Connect integration.
Backend FastAPI (Python) Fast enough for production, Python ecosystem for AI/ML integration, async support for LLM streaming.
Database PostgreSQL 15 + TimescaleDB TimescaleDB hypertables for health time-series queries. Row-level security for multi-tenant health data isolation.
AI AI API (LLM-based) Best-in-class instruction following for structured plan generation.
Knowledge ChromaDB + RAG Retrieval-augmented generation for cultural food and health research. Pre-built knowledge base per community.
Infrastructure AWS ECS Fargate Serverless containers. HIPAA-eligible services. Multi-AZ deployment. Auto-scaling for plan generation spikes.
Auth JWT + refresh rotation 15-min access tokens. 30-day refresh rotation. Biometric re-auth for sensitive health data screens.
