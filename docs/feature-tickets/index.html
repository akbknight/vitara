Feature Ticket List

This document translates the PRD into concrete, buildable work items organized into five milestones. Each ticket has an ID, priority, acceptance criteria, and dependencies. This is the starting point for sprint planning — not a frozen backlog. Tickets will be refined as discovery interviews produce sharper requirements.

Ticket Format

Each ticket contains the following fields:

FieldDescription
IDStable identifier: MODULE-NNN (e.g. AUTH-001, PLAN-003). Never reused even if the ticket is deleted.
PriorityMUST (required for MVP), SHOULD (strong preference for v1), COULD (nice-to-have in v1, acceptable to defer to v1.1)
Feature NameShort human-readable name, used in PR titles and sprint board cards
DescriptionWhat this ticket builds and why it matters
Acceptance CriteriaBulleted list of verifiable conditions that must all be true for the ticket to be considered done
DependenciesOther ticket IDs that must be complete before this one can be started or completed
Discovery Phase
MUST-priority tickets across M1 (Foundation), M2 (Adaptive Plan Engine), and M3 (basic coaching) define the MVP. Everything in M4 and M5 that is marked SHOULD or COULD is explicitly post-validation scope — build it only after the first 50–100 users have confirmed the core personalization hypothesis. Do not let roadmap excitement pull M4/M5 COULD tickets into the MVP sprint.

Roadmap Timeline

Milestone 1 — Foundation

M1 builds the account and health profile infrastructure that every other milestone depends on. No AI, no plans, no tracking — just solid auth, a complete profile model, and wearable data access from day one (even if the data isn't used until M2).

AUTH-001 MUST User Registration

Description: Create an account with email and password. Send email verification. Store in users table with is_verified=false. Block access to all non-auth routes until verified.

Acceptance Criteria:

  • POST /api/v1/auth/register accepts email + password; returns 201 with user_id
  • Password is bcrypt-hashed before storage; plaintext never logged
  • Verification email delivered within 60 seconds; link expires after 24 hours
  • Duplicate email returns 409 Conflict (not a 500)
  • All auth routes return JSON errors with consistent shape: {code, message}
  • Registration form on mobile validates email format + password strength (min 8 chars, 1 number) before API call

Dependencies: None (first ticket)

AUTH-002 MUST User Login + JWT

Description: Validate credentials and issue a 15-minute access token + 30-day refresh token. Refresh token stored in flutter_secure_storage. Implement automatic silent refresh on 401.

Acceptance Criteria:

  • Successful login returns {access_token, refresh_token, expires_in: 900}
  • Access token payload contains: user_id, subscription_tier, iat, exp
  • Refresh token stored in flutter_secure_storage (not SharedPreferences) — verified with unit test
  • Expired access token triggers silent refresh without user intervention; request is retried automatically
  • 5 consecutive failed logins lock the account for 30 minutes; user receives alert email
  • POST /auth/logout invalidates refresh token in Redis; subsequent refresh attempts return 401

Dependencies: AUTH-001

AUTH-003 SHOULD Biometric App Lock

Description: After the app is backgrounded for more than 5 minutes, require Face ID or fingerprint (or a 6-digit PIN as fallback) before showing any health data. Does not re-issue tokens — gates access to the stored refresh token.

Acceptance Criteria:

  • App detects background/foreground transitions and starts a 5-minute lock timer
  • Biometric prompt uses local_auth package; graceful fallback to PIN if biometric unavailable
  • 5 failed PIN attempts clears stored tokens and requires full login
  • Lock screen does not display any health data (black screen or VITARA logo only)

Dependencies: AUTH-002

PROFILE-001 MUST Health Onboarding Flow

Description: 7-screen onboarding flow that collects the full profile needed for plan generation: goal input, ethnicity + food culture, location (city/country), body rhythm (wake/sleep/peak energy time), dietary preferences and restrictions, basic health metrics (height, weight, age, known conditions). Saves incrementally — each screen writes to user_profiles; if the user exits mid-flow they resume from the last completed step.

Acceptance Criteria:

  • Each screen writes to user_profiles on "Next" tap; partial profile is never lost
  • Ethnicity selector allows multi-select (up to 4 ethnicities); includes: South Asian (Indian, Pakistani, Bangladeshi, Sri Lankan), West African, East African, Middle Eastern, East Asian, Southeast Asian, Latin American, European, Other with free text
  • Dietary preferences screen includes: vegetarian, vegan, halal, kosher, pescatarian, lacto-vegetarian, Jain; plus allergens: nuts, gluten, dairy, shellfish, eggs, soy
  • Wake/sleep time inputs use a time-wheel picker with timezone auto-detected from device
  • Total onboarding time under 4 minutes (tested with real user timing on pilot sessions)
  • Progress indicator shows current step / 7 throughout
  • Onboarding cannot be skipped entirely; users can tap "Skip for now" on optional fields only (health conditions, medications)

Dependencies: AUTH-001, AUTH-002

PROFILE-002 SHOULD Profile Edit

Description: Allow users to update any field from their onboarding profile after completing it. Changes to ethnicity, dietary preferences, wake/sleep time, or health conditions should flag the current plan as stale and prompt a plan refresh.

Acceptance Criteria:

  • All onboarding fields are editable from Profile settings screen
  • Changes to plan-impacting fields (ethnicity, dietary prefs, body rhythm) set plan_status = 'stale' and show a banner: "Your profile changed. Refresh your plan?"
  • Weight updates log a new entry in health_metrics (metric_type = 'body_weight') rather than overwriting the onboarding value
  • PATCH /api/v1/users/profile returns updated profile; no full page reload needed

Dependencies: PROFILE-001

PROFILE-003 MUST Account + Data Deletion (GDPR)

Description: Users must be able to request full deletion of their account and all associated health data. 30-day soft-delete period; hard deletion on day 31 covers all PostgreSQL rows, S3 objects, and ChromaDB embeddings.

Acceptance Criteria:

  • Delete account option accessible from Settings; requires password confirmation before submitting
  • User receives email confirming deletion request and 30-day timeline
  • During 30-day period: account is locked (cannot log in) but data is not yet deleted
  • On day 31: automated Celery task deletes all rows in users, user_profiles, health_metrics, meals, coaching_sessions, lab_results, user_integrations for that user_id
  • S3 objects (food photos, lab images) deleted by user_id prefix
  • Confirmation email sent when hard deletion completes
  • User can cancel deletion during the 30-day window via a link in the confirmation email

Dependencies: AUTH-001

NOTIF-001 MUST Push Notification Setup

Description: Firebase Cloud Messaging integration. Request notification permission on first open (after onboarding, not immediately on install). Store FCM token server-side for targeting.

Acceptance Criteria:

  • FCM token generated and stored in users table on permission grant
  • Permission request shown after onboarding step 3 (location), with a clear value explanation: "We'll send your daily health brief"
  • FCM token refresh handled automatically when FCM rotates the token
  • Test notification can be sent from admin panel to a specific user_id

Dependencies: AUTH-002, PROFILE-001

HEALTH-001 MUST HealthKit Integration (iOS)

Description: Request HealthKit permissions and read sleep analysis, HRV, resting heart rate, steps, and active energy on app launch. Sync to health_metrics via background task.

Acceptance Criteria:

  • Permission request screen explains exactly what data will be read and why, with plain-language descriptions (not Apple's default sheet alone)
  • If permission denied, app functions with gracefully degraded plan accuracy; shows "Connect Apple Health to improve your plan" prompt
  • Reads last 7 days of data on first connection; then incremental sync on each app open
  • Sync result POSTed to /api/v1/health-data/sync; deduplication by (user_id, time, metric_type) on server
  • No health data logged to console, analytics, or Sentry events

Dependencies: AUTH-002

HEALTH-002 MUST Health Connect Integration (Android)

Description: Android equivalent of HEALTH-001. Use the health Flutter package to read from Health Connect aggregator. Same data types, same sync endpoint, same deduplication logic.

Acceptance Criteria:

  • Health Connect permission request includes all required permission strings in AndroidManifest.xml
  • Fallback behavior if Health Connect not available on device (older Android): show "Health tracking limited" notice, proceed without wearable data
  • Same data quality and sync behavior as HEALTH-001
  • Tested on Android 13+ (Health Connect native) and Android 9–12 (Health Connect installable)

Dependencies: AUTH-002

Milestone 2 — Adaptive Plan Engine

M2 builds the core AI differentiation: taking the onboarding profile and generating a plan that is genuinely culturally calibrated, location-aware, and body-rhythm-aligned. This is the moment of proof. Every ticket in M2 is MUST.

AI Prompting Investment
PLAN-001 and COACH-001 both depend on well-engineered prompts. The quality of the generated plan is entirely a function of prompt quality — no amount of UI polish will rescue a plan that recommends grilled chicken and a 6am workout to a vegetarian night-owl in Birmingham. Invest 2–3 days in prompt engineering with real user profiles before building any plan display UI. Run the prompts manually via the Anthropic console with 20+ diverse profile inputs and evaluate the outputs against the "moment of recognition" standard described in the PRD.
PLAN-001 MUST Ethnicity-Based Meal Plan Generator

Description: Given the user's ethnicities and dietary preferences, generate a 7-day meal plan using the Claude API with culturally appropriate foods. The South Asian food database is built first; expansion hooks are in place for other cuisines.

Acceptance Criteria:

  • Generated plan includes real dish names from the user's cuisine (e.g. "Poha with peanuts and curry leaves" not "grain dish with legumes")
  • Macros for each meal are calculated and stored; totals match daily caloric target ± 10%
  • Dietary restrictions are enforced without exception (halal, vegetarian, allergens)
  • Plan is stored in PostgreSQL and retrievable from GET /api/v1/nutrition/meal-plan
  • Meal plan generation completes in under 30 seconds for 95% of requests
  • If generation fails (Claude API timeout), return the last cached plan + "Updating your plan" notice; never show a blank plan screen

Dependencies: PROFILE-001, AUTH-002

PLAN-002 MUST Location-Aware Exercise Planning

Description: Use the user's city and country to adapt exercise recommendations to local climate, season, and likely infrastructure availability. A user in Houston in July gets different outdoor advice than one in London in January.

Acceptance Criteria:

  • Exercise recommendations reference the user's city explicitly ("For Houston's summer heat...")
  • Climate inference is based on city + current month (no real-time weather API required in v1)
  • Gym availability inferred from country/city tier: major Western cities assume gym access available; recommendations offer home workout alternatives for all plans regardless
  • Exercise plan stored as part of the week's plan structure in PostgreSQL alongside the meal plan

Dependencies: PLAN-001 (same plan generation call)

PLAN-003 MUST Body Rhythm Scheduling

Description: Read usual_wake_time and usual_sleep_time from the user profile and use them to schedule meal windows, intermittent fasting windows (if applicable), and workout timing recommendations around the user's actual circadian cycle rather than a default 6am–10pm assumption.

Acceptance Criteria:

  • Meal times in the generated plan align with the user's wake time: first meal within 1 hour of wake time (or 2 hours for IF plans)
  • A user who wakes at 2pm receives an afternoon breakfast and a late-night dinner; the plan acknowledges the schedule without judgment
  • Peak energy window (user-provided in onboarding) is used to place the most demanding workout in that window
  • Ramadan mode: if user has flagged Ramadan active, sehri is scheduled 30 minutes before Fajr and iftar at Maghrib (using approximate prayer time calculation based on city + date)

Dependencies: PLAN-001, PROFILE-001

PLAN-004 MUST Plan Review Screen

Description: Display the generated 7-day plan to the user after onboarding step 6 (plan generation). The review screen must create the moment of recognition. Allow the user to adjust individual meals or workout suggestions before confirming.

Acceptance Criteria:

  • Plan displayed as a scrollable 7-day view with clear day headings and meal sections
  • Each meal shows: dish name, macro summary (cal / protein / carbs / fat), and meal type
  • Loading screen during generation shows progressive status messages: "Reading your food heritage..." → "Adapting to your timezone..." → "Building your first week..."
  • User can tap any meal to replace it ("Suggest something else"); triggers a targeted re-generation for that meal slot only
  • Confirm button stores the approved plan and navigates to the main app dashboard

Dependencies: PLAN-001, PLAN-002, PLAN-003

PLAN-005 MUST Plan Storage & Dashboard Display

Description: Store the confirmed plan in PostgreSQL and display it on the app's main dashboard as the default home screen. Today's meals and today's workout are the primary content surface of the home screen.

Acceptance Criteria:

  • Plan stored with status = 'active' in a plans table; GET endpoint returns current day's plan slice
  • Home screen shows: today's meal plan (3–5 meal slots), today's workout, and the daily brief (empty state until COACH-001 is built)
  • Plan survives app close and reopen without re-fetching (cached in Isar local DB)
  • Past days are accessible via a horizontal scroll calendar strip

Dependencies: PLAN-004

PLAN-006 SHOULD Plan Refresh

Description: Regenerate the plan weekly (automatic) or on user request. Refresh takes updated health data and any profile changes into account. The previous plan is archived, not deleted.

Acceptance Criteria:

  • Celery scheduled task triggers plan refresh every Sunday at 9pm user-local time
  • User can manually trigger a refresh from the Plan screen; refresh costs one "plan refresh" credit per month on Free tier (unlimited on Plus/Pro)
  • Previous plan stored with status = 'archived'; accessible from plan history screen
  • Refresh incorporates last 7 days of health metrics from wearable sync if available

Dependencies: PLAN-005, HEALTH-001

Milestone 3 — AI Coaching

M3 builds the ongoing relationship with the AI coach. The plan from M2 answers "what should I do this week?" — M3 answers "how am I doing?" and "I have a question." The daily brief is the habit-forming touchpoint; the chat interface is the depth layer.

COACH-001 MUST Daily Brief Generation

Description: Every morning (at user-local wake time + 30 minutes), a Celery task generates a personalized daily brief using the past 7 days of health context. The brief includes: yesterday's health snapshot, today's plan highlights, one actionable insight, and one motivational observation grounded in the user's actual data.

Acceptance Criteria:

  • Celery beat task runs at (wake_time + 30min) in user's timezone daily; handles timezone edge cases correctly
  • Brief generated using the system prompt template in app/core/ai/prompts.py with last 7 days of metrics injected
  • Generated brief cached in Redis with a 24-hour TTL; GET /ai/daily-brief returns cached version instantly
  • Brief displayed on home screen in a dedicated card above the plan; max 4 sentences
  • Push notification sent with brief headline at generation time (if NOTIF-001 complete)
  • If no health data available (wearable not connected), brief acknowledges this and uses profile-only context

Dependencies: PLAN-005, HEALTH-001 or HEALTH-002, NOTIF-001

COACH-002 MUST Chat Interface

Description: A real-time chat interface with the AI health coach. User messages are sent to the Claude API with a system prompt that includes the user's health context snapshot and the last 10 messages of conversation history. Responses stream to the UI.

Acceptance Criteria:

  • Chat interface uses the coaching chat bubble components from the Frontend Spec
  • Streaming response: text appears character-by-character as Claude streams tokens; no jarring full-message pop-in
  • Full conversation stored in coaching_sessions.messages JSONB after each exchange
  • Health context snapshot injected at session start: includes current plan, last 7 days of key metrics, user goals, profile summary
  • Free users see remaining message count ("2 of 3 messages today") with upgrade prompt when exhausted
  • AI never claims to be human; never provides specific medical diagnoses; prefixes medical-adjacent advice with appropriate disclaimer

Dependencies: PROFILE-001, HEALTH-001, PLAN-005

COACH-003 SHOULD Voice Mode

Description: Allow users to speak to their AI coach using the device microphone. Speech is transcribed via speech_to_text, sent to the Claude API as text, and the response is read aloud via flutter_tts.

Acceptance Criteria:

  • Voice mode toggled from within the chat interface via a microphone button
  • Transcription displayed in the user bubble so the user can verify what was sent
  • TTS reads the AI response at a comfortable pace (1.0x speed default, adjustable)
  • Voice mode is not the default; text chat is always available as the primary interface

Dependencies: COACH-002

COACH-004 SHOULD Suggested Questions

Description: Display 3 contextual question chips below the chat input, generated based on the user's current health state and plan. Tapping a chip pre-fills the message input. Regenerated daily based on fresh context.

Acceptance Criteria:

  • Suggestion chips are specific to the user's context (e.g. "Why is my HRV low this week?" not "How can I improve my health?")
  • Generated daily as part of the daily brief Celery task; cached alongside the brief
  • 3 chips shown; user can dismiss individual chips; chips reload after a message is sent

Dependencies: COACH-001, COACH-002

COACH-005 COULD Pinned Insights

Description: Users can long-press any AI message in the chat to pin it to their profile as a saved insight. Pinned insights appear in a dedicated "Your insights" section in the profile tab.

Acceptance Criteria:

  • Long-press gesture on AI bubble shows "Pin this insight" action sheet
  • Pinned insights stored in a separate table (user_id, coaching_session_id, message_index, content, pinned_at)
  • Maximum 20 pinned insights per user; older pins can be deleted

Dependencies: COACH-002

Milestone 4 — Nutrition & Tracking

M4 builds the logging layer: food, hydration, and health metric visualization. These features make VITARA a daily-use app rather than a one-time plan generator. The MUST tickets here are minimum for launch; the SHOULD/COULD tickets improve daily engagement but don't block the core value prop.

NUT-001 MUST Manual Food Logging

Description: Search the food database (USDA FoodData Central + Open Food Facts) and log individual foods with quantity. Calculate meal macros and store in the meals table. South Asian foods must be findable by their actual names.

Acceptance Criteria:

  • Search returns results from both USDA and Open Food Facts, deduplicated and ranked by relevance
  • "Dal makhani", "roti", "poha", "biryani", "chapati" all return accurate results as the top hit
  • Quantity input supports both gram-weight and common unit measures (1 chapati, 1 katori, 1 cup)
  • Logged meal macros appear on the home screen daily summary within 3 seconds of logging
  • Macro data stored in meals table; accessible via coaching context for AI-informed suggestions

Dependencies: AUTH-002, PLAN-005

NUT-002 SHOULD Barcode Food Scanner

Description: Use the device camera to scan a product barcode. Look up the barcode in Open Food Facts API. Auto-fill the food logging form with product nutrition data.

Acceptance Criteria:

  • Barcode scanner uses mobile_scanner Flutter package; real-time camera feed with targeting overlay
  • Successful scan auto-populates food name and nutrition data; user confirms quantity before logging
  • If barcode not found in Open Food Facts: show "Product not found — search manually" fallback
  • Works with EAN-13 (UK/EU) and UPC-A (US) barcodes

Dependencies: NUT-001

NUT-003 COULD Photo Food Recognition

Description: User takes a photo of a meal; GPT-4o Vision identifies the foods and returns estimated nutrition data. User confirms or adjusts before logging. Plus/Pro tier only.

Acceptance Criteria:

  • Photo upload via POST /api/v1/nutrition/photo; processed server-side (API key not exposed to client)
  • Response includes: identified food items, estimated quantities, total macros with confidence note
  • User can edit any field before confirming the log; confidence note displayed for low-confidence identifications
  • Photo deleted from S3 within 24 hours unless user explicitly saves the meal log with photo
  • Paywalled behind Plus/Pro; Free users see feature preview with upgrade prompt

Dependencies: NUT-001, SUBS-001

NUT-004 COULD Voice Food Logging

Description: "I just ate two chapatis and a bowl of dal makhani" — user speaks a meal description; NLP parses it into individual food items and quantities; user confirms before logging.

Acceptance Criteria:

  • Speech recognized via speech_to_text; transcription sent to Claude API for food extraction as a structured task (not a coaching call)
  • Extracted foods displayed as a list for confirmation before logging
  • Handles common South Asian food names and quantity descriptions ("one katori", "two rotis")

Dependencies: NUT-001, COACH-003

METRIC-001 MUST Health Dashboard

Description: Display this week's key health metrics — sleep duration, HRV, resting heart rate, daily steps, active energy — as a clean dashboard using fl_chart. Each metric shows current value + 7-day trend.

Acceptance Criteria:

  • Dashboard accessible from bottom nav "Health" tab
  • Each metric displayed as a HealthMetricCard (from Frontend Spec) with value, unit, trend arrow, and 7-day average
  • Trend lines rendered with fl_chart; 7-day bar or line chart for each metric
  • If no wearable data: dashboard shows "Connect Apple Health or Google Health to see your metrics" with a connect button
  • Semantic labels on all charts for screen reader accessibility

Dependencies: HEALTH-001 or HEALTH-002

METRIC-002 SHOULD Trend Analysis (7d / 30d / 90d)

Description: Allow users to switch between 7-day, 30-day, and 90-day time range views for each health metric. The TimescaleDB backend is already set up to support these queries efficiently.

Acceptance Criteria:

  • Time range toggle (7d / 30d / 90d) on each metric detail screen
  • 30d and 90d views use weekly aggregates to keep chart legible (daily data points on 7d view)
  • Backend query uses TimescaleDB time_bucket() for aggregation; must respond in under 500ms

Dependencies: METRIC-001

WATER-001 SHOULD Hydration Tracker

Description: Daily water intake goal (calculated from weight + activity level + climate) with quick-add buttons (200ml, 250ml, 350ml, 500ml). Progress ring on the home screen.

Acceptance Criteria:

  • Daily goal calculated at plan generation time; stored in user plan data
  • Quick-add buttons on home screen; custom amount input available via tap-and-hold
  • Today's intake stored in health_metrics (metric_type = 'water_ml', source = 'manual')
  • Hydration progress ring on home screen updates in real time on each log

Dependencies: PLAN-005

Milestone 5 — Polish & Launch

M5 makes the app shippable: subscription gates, empty states, contextual notifications, performance optimization, and App Store submission. None of these tickets add new features — they make existing features production-ready.

SUBS-001 MUST Subscription Paywall

Description: RevenueCat integration for in-app subscription. Free tier (3 AI messages/day, no plan refresh, no photo logging) vs. Plus ($12/mo, full coaching + plan refresh + photo logging). Paywall screen shown when free limit is reached.

Acceptance Criteria:

  • RevenueCat SDK initialized with correct project credentials for App Store and Play Store
  • Paywall screen shows Plus vs. Free comparison clearly; no dark patterns
  • Purchase flow uses native StoreKit (iOS) and Google Play Billing (Android) via RevenueCat
  • RevenueCat webhook updates subscription_tier in users table within 30 seconds of purchase
  • Trial period (7 days free Plus) configured in RevenueCat; auto-converts to paid after trial unless cancelled
  • Subscription tier checked server-side on all gated routes — never trust client-side tier claim only

Dependencies: AUTH-002, PROFILE-001

ONBOARDING-001 MUST Empty States & Tutorial

Description: Every screen that can be empty (no health data, no meals logged today, no coaching history) must have a helpful empty state that explains what the screen will show and how to get there. No blank screens.

Acceptance Criteria:

  • Health dashboard empty state: illustration + "Connect Apple Health to see your metrics" + connect button
  • Chat history empty state: "Start a conversation with your health coach" + suggested first questions
  • Meal log empty state: "Nothing logged yet today" + quick-add button + "Your plan suggests [today's first meal]" nudge
  • All empty state illustrations are culturally neutral (abstract, not food/person imagery)

Dependencies: All M1–M4 MUST tickets

NOTIF-002 SHOULD Contextual Reminders

Description: Scheduled push notifications for meal times, workout windows, and water intake — all scheduled relative to the user's actual plan and wake/sleep times, not a fixed 12pm reminder.

Acceptance Criteria:

  • Meal reminders sent 15 minutes before each planned meal time
  • Workout reminder sent at the scheduled workout window start
  • Hydration reminder sent every 2 hours between wake time and 2 hours before bedtime
  • All reminders are cancelable from notification settings; system respects quiet hours (no notifications within 1 hour of bedtime)

Dependencies: NOTIF-001, PLAN-005

PERF-001 MUST Performance Optimization

Description: Ensure the app opens fast and scrolls fluidly on mid-range devices. Health data loading must be paginated and lazy; local Isar cache must be the first data source for all home screen content.

Acceptance Criteria:

  • Cold start to home screen: under 3 seconds on a mid-range Android device (Pixel 6a equivalent)
  • Home screen content served from Isar local cache first; network refresh happens in background
  • Health metric list virtualized (ListView.builder, not Column with all widgets); no jank on 90-day history scroll
  • Images lazy-loaded with placeholder; no layout shift when images load
  • Flutter DevTools performance overlay shows no frames consistently above 16ms on home screen scroll

Dependencies: METRIC-001, PLAN-005

TEST-001 MUST Core User Journey Tests

Description: End-to-end integration tests covering the critical user journey: register → verify email → complete onboarding → generate plan → view plan → send one coaching message. Must pass on both iOS and Android simulators in CI.

Acceptance Criteria:

  • E2E tests written with flutter_test + integration_test package
  • Test covers: registration, email verification (stubbed), 7-screen onboarding, plan generation (mocked Claude API response), plan review, coaching chat (mocked response)
  • Tests run in GitHub Actions on every PR to main; PR cannot be merged if tests fail
  • Backend API integration tests cover: auth flow, profile CRUD, health data sync deduplication, plan generation endpoint, subscription webhook processing

Dependencies: All M1–M4 MUST tickets

RELEASE-001 MUST App Store Submission

Description: Prepare and submit VITARA to the iOS App Store and Google Play Store. Includes screenshots, metadata, privacy nutrition labels, and App Store Review preparation.

Acceptance Criteria:

  • App Store Connect: 6.5" iPhone screenshots, 12.9" iPad screenshots (if applicable), app description (250 words), privacy nutrition labels accurately reflecting all data collected
  • Google Play Console: feature graphic, phone screenshots, full description, data safety form
  • HealthKit usage description strings in Info.plist accurately describe each permission (App Review team reads these)
  • All third-party SDKs declared in App Store privacy labels (RevenueCat, Firebase, Sentry, Anthropic API)
  • Pre-launch security checklist (from Security & Access doc) 100% complete before submission

Dependencies: All MUST tickets across M1–M5, Pre-Launch Security Checklist complete

AI Prompt Templates

These three prompt templates are the starting point for app/core/ai/prompts.py. They should be treated as living documents — refined through testing with real user profiles. The JSON placeholder blocks represent data that is injected programmatically at runtime.

Prompt Engineering Before UI
Run these prompts directly in the Anthropic Workbench with 20 diverse user profiles before writing any Flutter code that displays plan output. If the prompts don't produce the "moment of recognition" for a vegetarian Gujarati in Washington DC, a night-shift nurse from Lagos in London, and a Mexican-American night owl in Houston — the UI is irrelevant. The prompts are the product.

Template 1 — Daily Brief Generation

You are VITARA, an adaptive health coach with deep knowledge of nutrition,
exercise science, and cultural food systems. Your role is to give brief,
specific, actionable daily health guidance.

Today's date: {current_date}
User timezone: {user_timezone}

USER PROFILE SUMMARY:
- Name: {first_name}
- Age: {age}, {biological_sex}
- Ethnicity/food heritage: {ethnicities}
- Location: {city}, {country}
- Health goals: {health_goals}
- Known conditions: {health_conditions}
- Usual wake time: {wake_time}, bedtime: {sleep_time}
- Dietary preferences: {dietary_preferences}

YESTERDAY'S HEALTH METRICS:
{health_metrics_json}

TODAY'S PLAN HIGHLIGHTS:
{todays_meals}
{todays_workout}

Write a daily brief in exactly 3–4 sentences. Tone: warm, specific, informed.
Do not use generic motivational language. Reference the user's actual data.
Acknowledge their cultural context when relevant (food names, timing practices).
End with one concrete, specific thing to do differently today based on their
metrics — not a generic tip, but one grounded in their data.

Do not use the user's name more than once. Do not use emojis.

Template 2 — Adaptive Meal Plan Generation

You are VITARA's nutrition planning engine. Your task is to generate a
culturally accurate, nutritionally balanced 7-day meal plan.

USER PROFILE:
- Ethnicities/food heritage: {ethnicities}
- Location: {city}, {country} (current season: {season})
- Wake time: {wake_time} | Bedtime: {sleep_time} | Peak energy: {peak_energy_time}
- Dietary type: {dietary_preferences}
- Dietary restrictions/allergies: {dietary_restrictions}
- Health conditions: {health_conditions}
