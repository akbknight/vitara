Security & Access Document

Health data is the most personal information a user can share with an app. This document specifies VITARA's authentication model, data classification scheme, access controls, and the edge cases that must be handled before any code ships to users.

Executive Summary

VITARA stores ethnicity, health conditions, medications, sleep patterns, dietary practices, and fitness data. This is among the most sensitive personal data a consumer app can hold — not because of a legal framework, but because of the real harm that could result from a breach: discriminatory use by insurers, employers, or immigration authorities; social harm from exposure of religious dietary practices; and the deeply personal nature of health struggles. Our security posture is built on four principles: user data is inaccessible to anyone other than the user (enforced at the database layer, not just the application layer); health data never leaves VITARA's infrastructure to third parties; every sensitive action is auditable; and security degradation fails closed (a broken token means no access, not degraded access).

Health Data Note
VITARA is not a medical device and does not claim HIPAA compliance. We are not a covered entity under HIPAA, and our users are not patients receiving medical treatment. However, we voluntarily apply PHI-adjacent security practices — AES-256 at rest, TLS 1.3 in transit, row-level security, AWS KMS for key management, and no third-party health data sharing — because the data we hold warrants that level of care regardless of legal obligation. This distinction must be communicated clearly in our privacy policy and terms of service.

Authentication Design

VITARA uses a stateless JWT architecture with short-lived access tokens and rotating refresh tokens. The design is optimized for mobile clients where background token refresh is possible and the blast radius of a stolen token must be minimized.

Access Token

15-minute expiry. Signed with HMAC-SHA256. Contains: user_id, subscription_tier, issued_at, expires_at. Short expiry means a stolen access token is useless within 15 minutes without the refresh token. Stored in memory (Riverpod state) on mobile — never written to disk.

Refresh Token

30-day expiry, rotation on each use. Stored in flutter_secure_storage (Keychain on iOS, Keystore on Android). When a refresh request arrives, the server issues a new refresh token and invalidates the old one in Redis. If the old token is presented again (stolen token reuse), both are invalidated and the user is logged out.

Biometric Lock

App-level Face ID / fingerprint. A separate 4-hour app-lock PIN or biometric challenge gates re-entry into the app after it's been backgrounded. This is enforced client-side using flutter_local_auth; it does not issue new tokens — it gates access to the stored refresh token.

Email Verification

Required before first plan generation. Registration creates a user with is_verified=false. All routes except /auth/* require is_verified=true. Verification link expires after 24 hours. Resend is rate-limited to 3 per hour per email.

Token Storage — Non-negotiable
Never store JWT tokens (access or refresh) in Flutter's SharedPreferences, AsyncStorage (React Native pattern), or any location accessible to other apps. Always use flutter_secure_storage which maps to iOS Keychain and Android Keystore. These storage locations are encrypted by the OS and are inaccessible to other applications. A token in SharedPreferences can be read by any app with file system access on a rooted device. A token in Keychain cannot.

Row-Level Security

PostgreSQL row-level security (RLS) is enabled on all user data tables. This means the database itself enforces that a query for health metrics can only return rows where user_id = current_setting('app.current_user_id') — even if the application layer makes a programming error and omits the user_id filter. RLS is the last line of defense against cross-user data leakage.

-- Example RLS policy on health_metrics
ALTER TABLE health_metrics ENABLE ROW LEVEL SECURITY;

CREATE POLICY user_isolation ON health_metrics
  USING (user_id = current_setting('app.current_user_id')::uuid);

-- Set in every database session after authentication:
SET LOCAL app.current_user_id = '<authenticated-user-uuid>';

The same RLS policy pattern is applied to: user_profiles, health_metrics, meals, coaching_sessions, lab_results, and user_integrations. The users table has a more restricted policy: users can read their own row but not write subscription_tier directly (only the RevenueCat webhook handler, running as a privileged role, can update that column).

No admin role can query individual health data through the application API. Admin tooling accesses only the analytics database, which contains aggregated, anonymized data. Any legitimate support access to individual user data (e.g., investigating a bug report) requires explicit approval, goes through a separate read-only console with full CloudTrail audit logging, and the user is notified.

Error Handling Guide

Error TypeHTTP StatusUser-facing MessageDeveloper Action
Invalid credentials 401 "Email or password is incorrect." Increment failed_login_count for the email; do not specify which field is wrong (enumeration prevention)
Expired access token 401 with code: TOKEN_EXPIRED Silent — mobile client intercepts 401 and attempts refresh automatically Client must retry with refresh token before showing any error to user
Expired/invalid refresh token 401 with code: SESSION_EXPIRED "Your session has expired. Please sign in again." Client clears stored tokens, navigates to login screen; no automatic retry
Missing subscription 403 with code: SUBSCRIPTION_REQUIRED "This feature requires a Plus subscription. Upgrade to continue." Include feature_name in the error body so the client can show a targeted paywall
Rate limit exceeded 429 "You've reached your request limit. Please wait a moment." Include Retry-After header with seconds until limit resets; log for abuse monitoring
Accessing another user's data 404 (not 403) "Not found." Return 404 not 403 — do not confirm the resource exists; log as a security event in CloudWatch
Email not verified 403 with code: EMAIL_NOT_VERIFIED "Please verify your email address to continue. Check your inbox." Offer resend link; rate-limit resend to 3 per hour per email
AI service unavailable 503 "Your health coach is temporarily unavailable. Try again in a moment." Retry with exponential backoff (1s, 2s, 4s); after 3 failures, return cached last response + unavailability message
Wearable integration token expired 400 with code: INTEGRATION_REAUTH_REQUIRED "Your [Fitbit] connection needs to be reconnected. Tap to fix." Set integration status = 'reauth_required'; trigger push notification; do not silently fail sync
Malformed input 422 Field-level validation errors from Pydantic Return structured error body with field names and constraint violations; never expose stack traces
Internal server error 500 "Something went wrong on our end. We've been notified." Sentry captures exception with full context; do not include stack trace in response body

Privacy Considerations

  • No third-party health data sharing. Health metrics, meals, coaching conversations, and profile data never leave VITARA's infrastructure. They are not sold, rented, or shared with advertisers, data brokers, insurance companies, employers, or government agencies.
  • AI processing transparency. When user health data is sent to the Claude API, this must be disclosed in the privacy policy. Anthropic's API terms prohibit using API inputs to train models, but the privacy policy must state explicitly that health data is sent to a third-party AI provider for processing.
  • Full data export (right to portability). Users can request a complete export of all their data: profile, health metrics, meals, coaching sessions, and lab results. The export is generated as a JSON file and delivered via a signed S3 URL. Export requests are processed within 72 hours.
  • Right to deletion. Account deletion requests initiate a 30-day soft-delete period (data is not accessible but not yet purged). Hard deletion occurs on day 31: all PostgreSQL rows, S3 objects (food photos, lab results), and ChromaDB embeddings containing the user's ID are permanently deleted. A confirmation email is sent when deletion completes.
  • GDPR / CCPA applicability. VITARA serves users in the UK and EU (GDPR) and California (CCPA). We are a data controller. The privacy policy must include: lawful basis for processing (contract performance + legitimate interest), data subject rights (access, rectification, erasure, portability), data retention periods, and a DPA contact.
  • Analytics anonymization. Any usage analytics (feature engagement, session lengths, plan completion rates) are anonymized before storage in the analytics database. No personal identifiers appear in any analytics pipeline.

Health Data Security Specifics

The medications JSONB column in user_profiles is encrypted at the column level using pgcrypto's pgp_sym_encrypt with a key derived from AWS KMS. This means even a direct database query by a DBA cannot read medication names without the KMS key. The same column-level encryption applies to the raw OCR text from lab result images before it is processed.

Food photos uploaded for AI recognition are processed by GPT-4o Vision and then deleted from S3 within 24 hours unless the user explicitly saves the log entry. Lab result images are stored with a 1-year retention policy and accessed via 30-minute signed URLs only. Both bucket types have S3 Object Lock enabled on production (WORM — write once, read many) for a 72-hour grace period, preventing accidental or malicious immediate deletion.

Abuse Prevention

  • Auth endpoint rate limiting: POST /auth/login and POST /auth/register are limited to 10 requests per IP per minute via AWS WAF. After 5 failed login attempts for a single email within 10 minutes, the account is temporarily locked for 30 minutes and the user receives an email alert.
  • AI chat rate limiting: Free users are limited to 3 AI messages per day. Plus/Pro users are limited to 100 messages per day (well above normal usage; exists to prevent API cost abuse). Per-user token budgets enforce a monthly cap.
  • Device fingerprinting on suspicious login: If a login comes from a new device + new country combination, require email re-verification before issuing tokens. Use a lightweight fingerprint (IP, user-agent hash, timezone offset) stored with the session.
  • Upload size limits: Food photos capped at 10MB; lab result images capped at 25MB. Enforced at the API layer before S3 upload. MIME type validation: only image/jpeg and image/png accepted for both.
  • SQL injection: All database access through parameterized queries via SQLAlchemy ORM. No raw SQL string concatenation anywhere in the codebase.

Edge Cases Checklist

These 25 edge cases must be handled in code — tested and verified — before any public launch. Each represents a real failure mode observed in health apps.

Authentication
  • Refresh token presented after account deletion — must reject with 401
  • Two devices simultaneously refresh the same token — first wins; second must re-authenticate
  • User changes email — old refresh tokens must be invalidated
  • User changes password — all existing sessions must be terminated
  • Biometric authentication fails 5 times — app must lock and require PIN
  • JWT issued with wrong subscription_tier — always re-read tier from DB, not JWT claim, for gated routes
Health Data
  • Wearable sync sends duplicate timestamps — deduplicate by (user_id, time, metric_type) on UPSERT
  • User revokes HealthKit permission mid-session — handle gracefully, show "reconnect" prompt
  • Health metric value is null or out of range (e.g. HR=0) — reject at validation layer, never store
  • TimescaleDB partition missing for future time (clock skew) — write still succeeds, chunk created automatically
  • User logs meal with future timestamp — accept but flag; affects daily calorie calculations
AI & Plans
  • Claude API returns malformed JSON in plan generation — retry once with explicit JSON schema instruction; if fails again, return last cached plan with "updating" message
  • User's profile is empty (skipped onboarding) — block /ai/generate-plan with 400 ONBOARDING_REQUIRED
  • Context window too large (user with 6 months of history) — implement sliding window: last 7 days of metrics + weekly aggregates beyond that
  • User sets both vegetarian and "I eat chicken" preferences — detect contradiction in validation layer; ask user to clarify at profile edit
Subscriptions & Data
  • RevenueCat webhook arrives out of order (renewal before expiry) — always take the event with the latest original_purchase_date, not arrival order
  • User requests data export during deletion grace period — still process export; user has right to their data until hard delete
  • S3 signed URL expires before user opens it — regenerate on-demand; never cache signed URLs client-side beyond 30 minutes
  • User uploads non-food image for food recognition — GPT-4o Vision returns "no food detected"; handle gracefully, prompt to retake
  • App crashed mid-onboarding — profile is partial; resume from last completed step; never generate a plan from an incomplete profile
  • User in a country where the app isn't available (future) — return 403 GEO_RESTRICTED with clear message; do not silently fail
  • Time zone change (user moves country) — update timezone in profile; recalculate all scheduled Celery tasks for that user
  • Ramadan mode: user sets Ramadan active — meal type options shift to sehri/iftar; caloric targets adjust for compressed eating window
