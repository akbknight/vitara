Technical Architecture Document

This document describes the full technical stack for VITARA: why each technology was chosen, how the system components interact, the database schema, AI integration approach, and deployment topology. It is a reference for engineering decisions, not a build order — the actual sequence of implementation is tracked in the Feature Tickets doc.

Executive Summary

VITARA's technical stack is built around three non-negotiable requirements: native mobile health data access, real-time AI coaching with deep health context, and secure storage of PHI-adjacent data. The mobile layer is Flutter 3.x — a single codebase that gives us HealthKit on iOS and Health Connect on Android without two separate engineering tracks. The backend is FastAPI on Python, chosen because the entire AI and data science ecosystem lives in Python and async FastAPI handles the mix of low-latency API calls and longer AI generation tasks efficiently. Health metrics are stored in PostgreSQL with TimescaleDB — a hypertable extension that makes time-series queries over months of health data fast and cheap without moving to a purpose-built time-series database. The AI layer is an LLM API for all coaching and plan generation, with ChromaDB providing RAG retrieval from the cultural food and health knowledge base. All of this runs on AWS, with ECS Fargate keeping infrastructure management overhead low while preserving the ability to scale horizontally when user volume demands it.

Phase 1 Simplification
During customer discovery and early private beta, the full AWS ECS + RDS + ElastiCache stack is over-engineered and expensive to operate. In Phase 1, use Railway or Render for FastAPI hosting, a managed PostgreSQL instance (Railway or Supabase) with the TimescaleDB extension enabled, and Upstash for Redis. Migrate to full AWS infrastructure when you have 500+ active users or a paying enterprise customer. The FastAPI code is identical in both environments — only the connection strings change.

System Architecture Diagram

TimescaleDB Note
The health_metrics table is a TimescaleDB hypertable partitioned by time. This means a query like "give me all HRV readings for user X in the last 30 days" automatically scans only the relevant time partitions rather than the entire table. Without this, as each user accumulates months of wearable data (sleep, steps, HRV, heart rate — potentially 500+ rows per user per day), range queries degrade catastrophically. TimescaleDB is a PostgreSQL extension — it runs on the same RDS instance with no separate infrastructure, and all existing Postgres tooling, backups, and row-level security apply unchanged.

AI Integration Layer

All coaching intelligence flows through app/core/ai/coach.py. Every call to the AI API is structured as follows:

  1. Context assembly: The coach module queries the database for the user's profile, last 7 days of health metrics (sleep, HRV, steps, meals), current plan, and any flagged alerts. This context is serialized to a structured JSON block that becomes part of the system prompt.
  2. RAG retrieval: rag.py queries ChromaDB with an embedding of the user's query or the day's health context to retrieve relevant cultural food knowledge, metabolic health guidelines, and exercise science references. Retrieved passages are appended to the system prompt as a "knowledge context" block.
  3. Prompt construction: prompts.py contains template functions for each session type. The daily brief template injects yesterday's metrics + today's plan. The plan generation template injects the full onboarding profile. The chat template injects a sliding window of the last 10 messages plus the current health snapshot.
  4. API call: The assembled messages array is sent to the AI model with a max_tokens ceiling of 1,500 for chat responses and 4,000 for plan generation. Temperature is 0.7 for coaching chat and 0.3 for plan generation (more deterministic).
  5. Response parsing and storage: The response is parsed (plan generation uses structured output with JSON blocks); the full exchange is appended to the coaching_sessions record along with the token count.

The ChromaDB collection contains: the VITARA cultural food database (South Asian dishes with macros, cooking methods, and ingredient variants), curated metabolic health guidelines for the conditions most common in the beachhead segment (T2DM risk, PCOS, hypertension), and exercise science references for body-rhythm-aligned training. Documents are chunked at 512 tokens with 64-token overlap and embedded using OpenAI's text-embedding-3-small.

Deployment Architecture

Production runs entirely on AWS. The API runs as ECS Fargate tasks behind an Application Load Balancer. Celery workers run as a separate Fargate task definition sharing the same ECR image. RDS PostgreSQL is Multi-AZ with automatic failover; TimescaleDB extension is enabled at RDS provision time. ElastiCache Redis runs in cluster mode with one replica. All ECS tasks run in private subnets; only the ALB has a public-facing security group. S3 buckets have no public access; all object reads go through CloudFront with signed URLs. AWS WAF is attached to CloudFront and the ALB, with rate limiting rules (100 req/min per IP on auth endpoints, 1000 req/min general).

CI/CD: GitHub Actions builds the Docker image on merge to main, pushes to ECR, and triggers an ECS rolling deployment. Database migrations run as a one-off ECS task before the new API version takes traffic.

Scaling Notes

  • API layer: ECS Fargate auto-scales on CPU > 70%; each task handles ~200 concurrent connections with FastAPI's async workers. Horizontal scaling is trivial.
  • Celery workers: Daily brief generation has a predictable spike at local morning time per user's timezone; Celery tasks are time-zone-aware and distributed throughout the UTC day window. Scale worker count by queue depth metric.
  • TimescaleDB: Chunk compression kicks in for partitions older than 7 days; data retention policy deletes raw wearable samples after 1 year (aggregated summaries retained indefinitely). These policies keep the health_metrics table manageable at scale.
  • AI API costs: Monitor tokens_used per user per month. Implement per-user token budgets that throttle AI requests gracefully (return cached last response + "updating shortly" message) to prevent runaway costs from heavy users.

Monitoring & Logging

Three layers of observability:

  • CloudWatch: ECS container metrics (CPU, memory, task count), RDS metrics (connection count, query latency, replication lag), ElastiCache metrics (memory usage, evictions), ALB access logs. Alarm on: API P99 latency > 2s, error rate > 1%, RDS connections > 80% of max.
  • Sentry: All unhandled exceptions in FastAPI and Flutter clients. Attach user ID and request ID to every event for correlation. Monitor for AI-specific errors (AI API timeouts, malformed JSON in plan generation responses).
  • CloudTrail: Full audit log of all AWS API calls. Required for any future SOC 2 or HIPAA alignment. Specifically: KMS key usage events (any access to encrypted OAuth tokens is logged), S3 access for lab result images, IAM role assumption events.
